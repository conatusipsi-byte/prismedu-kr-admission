#### 3단계: 분류

##### ✅ 잘 잡힌 것
- `tracks.{jeongsi_na, susi_comprehensive}` 동시 보유 (정시인데 학종 트랙 있음) — Partial<Record> 구조 OK
- 학과 단위 `Department` 도큐먼트 분리 (모집단위 명칭이 학과여서 자연스러움)
- 학종 단계별 평가 (1단계 100% + 2단계 합산) — `AdmissionStage[]` 매핑 가능
- 영역별 `scoreType: "standard"` — 영역마다 다른 score type 보유 가능 구조

##### ⚠️ 어색하지만 들어가긴 함
- **영어/한국사 감점**: `gradeMap` 필드에 음수 값 저장. 의미는 "감점"인데 필드명은 환산점수처럼 양수 가정. polarity 운영 매뉴얼 명시 필요.
- **정시에서 교과평가**: `AdmissionStage.components.schoolRecord`로 매핑은 가능하나, 정시(`jeongsi_*`)에 schoolRecord가 들어가는 의미 부조화. `notes`에 보강.
- **반영비율 합 ≠ 100** (국 100 + 수 120 + 탐 80 = 300): 우리 검증식 `합=100` 가정과 맞지 않음. 비율을 비율 그대로 보유 + 정규화는 매칭 시 수행.
- **모집단위 내 상대 normalize**: 절대 점수 모델이 아님. 사용자 점수 환산 시 모집단위 분포가 필요. `bonusByCourse`나 별도 필드 없음 — `notes`에 적힘.

##### 🔴 못 담는 정보
- **수능 응시영역기준**: 자연계 미적분/기하 필수, 인문계 화법과작문 가능 — `CsatMinimum`은 합산 모델만 가정. 응시 자격 자체는 별도 필드 없음.
- **과탐 조합 조정점수** (Ⅰ+Ⅰ vs Ⅰ+Ⅱ vs Ⅱ+Ⅱ): `bonusByCourse`는 과목 단위, **조합 단위** 가산점은 X.
- **수능 성적 환산식 (상대 normalize)**: 모집단위 최고/최저점 기반 환산 — 절대 점수 모델로 변환 불가. `notes`에 식 적어두는 정도.
- **자유전공학부 "소속 유지" 옵션**: broadcast 모델은 sub 학과 선택 가정인데, 자유전공으로 졸업 가능성 표현 X.
- **건축학과 전공 배정 60% 룰** 등 학과 내 분배 정책.

##### 💡 도메인 인사이트
- **정시인데 학생부 정성평가**: 서울대 정시는 교과평가가 모든 모집단위 지원자에게 적용됨. "정시=수능만"이라는 우리 가정이 깨짐.
- **모집단위 내 상대평가가 점수 산출의 기본**: 절대 점수 비교가 아니라 그 학과 지원자 분포 안에서의 상대 위치. 사용자에게 "내 점수는 약 X점"이라는 답변이 어려움 — P-002 정직성 원칙과 일관.

- 일괄 선발 = `stages.length === 1`로 자연스럽게 표현
- simple_sum/simple_avg 분류기 정확 동작

##### ⚠️ 어색하지만 들어가긴 함
- 의대 vs 인문/자연 차등 수능최저 — `originalText` 한 덩어리로 담으면 conditional, 모집단위(의대 학과)별로 admissions/{year} 분리하면 simple_avg. **학과 단위 분리가 더 정확** — 현 스키마 정책에 부합.
- 학년별 반영비율 없음 — 우리 `SchoolRecord.gpaByTerm[]`은 학년별 단위 가능. "학년 가중치 없음"은 `gpaByTerm`을 모두 동일 가중치로 처리하는 의미. 표현 가능.

##### 🔴 못 담는 정보
- **대학별 학생부 환산표** (1등급=100, 2=96, ...): 학교마다 다름. 사용자 입력 등급 → 그 대학 점수로 환산하려면 표 보유 필수. 현재 `Department`·`AdmissionTrack` 어디에도 자리 없음.
- **변환석차등급 산출식** (성취도 A/B/C → 분포비율 기반): 진로선택 처리 방식이 학교별 다름. 우리 `absoluteDistribution: { A,B,C }`로 raw 보관은 OK 지만, **환산 로직** 자체는 못 담음.
- **수강자 수 조정등급** (수강자 1명 → 조정등급 4 등): 학교별 룰.
- **직업탐구 인정/미인정** 명시 필드 없음.

##### 💡 도메인 인사이트
- **학교추천의 일괄 선발 + 학생부 90% + 서류 10%** 구조는 내신 비중이 압도적. 학생부 환산표가 합격 추정의 핵심. 환산표 부재 시 우리 매칭 정확도 크게 떨어짐 — **P0 보강 후보**.
- **고려대는 학년별 가중치 없음**, 다른 대학은 있을 수 있음. `SchoolRecord` 모델은 학년별 단위 그대로 두되, 학교별 환산 시 학년 가중치 정책을 학과 단위에 보유.

### 2.2 광역모집 학과 단위 1차 키 정책 영향

`docs/schema.md` §2.2가 명시한 "학과 단위 = 1차 키" 원칙은 **유지 가능하지만, 광역모집·통합모집·계열모집의 비중이 우리가 가정한 것보다 높음**. 6개 PDF 중 4건(서울대 수시·정시, 외대 수시)에서 broadcast 모집단위가 다수 등장.

**영향**:
- broadcast 모집단위는 그 자체가 "학과"로 도큐먼트화 (`unitType="broadcast"` + `subDepartments[]`)
- 합격 추정 시 broadcast 학과의 합격선이 sub 학과 평균 vs 최저 vs 최고 중 어느 것인지 정책 결정 필요 (`docs/migration.md` §6 추가 함정 참조)
- broadcast 학과의 합격선 데이터는 입학 후 1~2년 후 sub 학과 분포가 결정되므로, `prevYearResult`에 sub 학과별 분포를 별도 보유하는 옵션 검토

→ 1차 키 정책 자체는 변경 불필요. broadcast 처리 정책(P-009 후보)을 신규 등록 권장.

```ts
   *  - "average":  subDepartments 평균 합격선 노출
   *  - "minimum":  가장 낮은 합격선 노출 (보수적)
   *  - "median":   중앙값
   *  - "per_sub":  분리 노출 (UI에서 sub 학과별로) */
  broadcastEstimationPolicy?: "average" | "minimum" | "median" | "per_sub";
  /** broadcast 모집단위에 입학 후 학과를 선택하지 않을 수 있는지 (자유전공 only) */
  broadcastAllowStay?: boolean;
}
```

## 5. 정책 후보 (policy.md 추가 권장)

### P-009 후보 — broadcast 모집단위 합격선 정책

- **상태**: dev-internal (실제 운영 데이터 누적 후 결정)
- **선택지**:
  - A. 평균 (균형, 운영 단순)
  - B. 최저 (보수적, 사용자 안전)
  - C. sub 학과별 분리 노출 (정확, UI 복잡)
- **결정 시점**: 시즌 1회 운영 후 사용자 만족도 + 표본 분포 기반

### P-010 후보 — 영어/한국사 감점 polarity

- **결정사항**: `ReflectionRatio.english.gradeMap` 의 값은 **양수=가산, 음수=감점** 으로 통일
- **이유**: 학교마다 환산점수(가산) vs 감점 모델이 다름. 단일 polarity로 처리하지 않으면 매칭 알고리즘 분기 폭증
- **운영 영향**: 모집요강 텍스트 "감점" 표현 시 ETL 파서가 음수로 변환

### P-011 후보 — 정시 + 학생부 정성평가 처리

- **결정 필요**: 서울대 정시처럼 학생부평가가 정시에 들어가는 학교 처리
- **선택지**:
  - A. `jeongsi_*` 트랙에도 `components.schoolRecord` 자유 사용 허용 (현재)
  - B. 별도 트랙 종류(`jeongsi_with_record_eval`) 신설
  - 권장 A — 단순, ETL 부담 적음

### P-012 후보 — 변환점수 ETL 2단계 운영

- **결정사항**: 모집요강 ETL은 매년 7~9월 1차 (변환표 미정), 12월 2차 (수능 후 변환표 추가) 운영
